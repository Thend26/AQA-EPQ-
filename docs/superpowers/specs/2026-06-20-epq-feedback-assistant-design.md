# EPQ 营地反馈助手设计规格

## 1. 项目目标

为唯寻 EPQ 营地助教建立一个可部署为网址的个人工作台，用于管理 1–10 名学生、记录每日成果，并结合学生前几天的成果、过程观察和历史反馈，生成面向家长或营地管理方的不少于 50 字的反馈。

首版依据 AQA Level 3 Extended Project Qualification 的核心能力维度组织记录和反馈，接入 DeepSeek，并支持中文、英文或中英双语输出。

## 2. 用户与使用场景

- 首版只有一名助教用户，通过邮箱登录个人空间。
- 每天为每名学生录入一次结构化记录。
- 用户可先让 AI 生成反馈，再通过直接对话要求调整语气、长度、语言和关注重点。
- 反馈确认后保存至学生历史，并自动成为下一次生成时的上下文。
- 反馈主要供家长或营地管理方阅读，语气应专业、具体、克制，不使用无证据的夸奖或绝对判断。

## 3. 技术方案

采用 Next.js、Supabase、Vercel 和 DeepSeek：

- Next.js 提供响应式网页、服务端接口及登录后的工作台。
- Supabase Auth 提供个人邮箱登录。
- Supabase Postgres 保存学生、每日记录、反馈版本和对话记录。
- Supabase Row Level Security 按用户 ID 隔离数据。
- Vercel 托管网页和服务端接口。
- DeepSeek API 由 Next.js 服务端调用。
- DeepSeek API Key 只存在于 Vercel 环境变量中，不发送到浏览器。

## 4. 页面结构

### 4.1 登录页

- 邮箱和密码登录。
- 首版不开放公开注册；账号由部署者预先创建或通过白名单限制。
- 登录失败时显示明确错误，不丢失已填写的登录信息。

### 4.2 工作台

采用三栏布局：

1. 左栏为学生列表，显示姓名、当前营地天数、最近阶段和今日是否已记录。
2. 中栏为选中学生的当日记录，包括 AQA 能力概览和结构化表单。
3. 右栏为 AI 对话、反馈草稿、语言切换及保存操作。

窄屏设备改为纵向布局，学生列表、记录表单和 AI 助手依次排列。

### 4.3 学生档案

每名学生保存：

- 显示姓名或匿名称呼；
- 年级；
- EPQ 研究题目；
- 营地开始日期；
- 背景备注；
- 当前关注点；
- 创建者用户 ID。

默认建议使用“林同学”等称呼，避免保存与反馈无关的敏感个人信息。

### 4.4 每日记录表单

结构化字段包括：

- 日期和营地天数；
- 今日完成成果；
- 成果证据或数量信息；
- 遇到的困难；
- 明日计划；
- 助教过程观察；
- 行为标签，例如主动提问、按时完成、需要提醒、能回应建议、合作表现突出；
- AO1–AO4 当日观察等级或备注。

表单在输入后自动保存为草稿。用户离开页面、网络中断或 AI 调用失败时，已输入内容仍应保留。

## 5. AQA EPQ 反馈框架

系统使用以下四个维度辅助组织观察，不直接伪装成正式评分：

- AO1 Manage：目标设定、计划、时间管理、过程推进和对变化的处理。
- AO2 Use resources：资料查找、选择、可信度判断、引用和资源运用。
- AO3 Develop and realise：研究方法、分析、论证、问题解决和成果形成。
- AO4 Review：对过程与成果的评价、局限识别、反思和后续改进。

AI 必须把判断建立在用户录入的事实之上。资料不足时应使用“目前观察到”“仍需继续关注”等审慎表达，不得编造成果、分数或能力结论。

## 6. 连续反馈生成

### 6.1 输入上下文

每次生成时，服务端组合：

- 学生基础档案；
- 当天结构化记录；
- 最近若干天的每日记录；
- 最近若干条已保存反馈；
- 前一日计划及其今日完成情况；
- 用户在当前反馈会话中的修改要求；
- AQA AO1–AO4 写作规则；
- 目标语言和目标读者。

首版优先使用最近 5 天完整记录。若历史更长，则附加系统生成的学生阶段摘要，控制提示词长度。

### 6.2 默认反馈结构

反馈应自然成段，覆盖：

1. 当天具体完成的成果；
2. 与前几天相比的进展、延续问题或计划落实情况；
3. 与 AQA 能力维度相关的过程判断；
4. 下一步具体、可执行的建议。

反馈不得少于 50 个中文字符；英文模式不得少于 50 个英文单词。双语模式分别生成中文和英文内容，不要求机械逐句翻译，但事实和判断必须一致。

### 6.3 AI 对话

用户可以直接要求：

- 更正式或更温和；
- 强调进步但避免夸大；
- 加入与昨天的比较；
- 缩短或扩写；
- 改为中文、英文或双语；
- 更突出某个 AQA 维度；
- 删除不适合向家长展示的内部观察。

每次修改基于当前草稿和当前会话，不重新丢弃上下文。

## 7. 质量检查与归档

反馈保存前执行以下检查：

- 达到所选语言的最低长度；
- 至少包含一项具体成果或证据；
- 至少包含一项下一步建议；
- 不包含系统无法从记录中支持的分数、排名或绝对判断；
- 双语模式下两种语言的关键事实一致；
- 学生、日期和每日记录关联正确。

未通过时显示具体原因，并允许用户继续对话修改。用户确认后，将最终文本、语言、使用的记录 ID、生成时间和版本保存到数据库。

## 9. 安全与隐私

- 所有业务表启用 Row Level Security。
- 查询、插入、修改和删除均要求 `owner_id = auth.uid()`。
- 服务端也校验当前用户是否拥有目标学生和记录，不依赖前端传入的所有权信息。
- DeepSeek API Key 只保存在服务端环境变量。
- 发送给 DeepSeek 的内容限制为生成反馈所需字段。
- 日志不得记录 API Key、完整认证令牌或不必要的学生信息。
- 删除学生时需二次确认，并明确会删除其每日记录、反馈与对话。

## 10. 异常处理

- AI 请求失败：保留表单和当前草稿，显示可理解的错误并提供重新生成。
- DeepSeek 返回格式异常：服务端尝试解析并校验；失败时不覆盖现有草稿。
- 数据保存失败：保留本地编辑状态并显示重试按钮。
- 登录过期：提示重新登录；未保存的表单草稿暂存于浏览器，登录恢复后可继续保存。
- 历史记录为空：按第一天模式生成，不声称存在趋势。
- 当天信息过少：提示用户补充事实，不允许 AI 用空泛内容代替证据。

## 11. 首版范围

首版包含：

- 个人邮箱登录；
- 1–10 名学生管理；
- 每日结构化记录和自动草稿；
- AQA AO1–AO4 观察框架；
- DeepSeek 生成和连续上下文；
- AI 对话修改；
- 中文、英文、双语切换；
- 质量检查和历史归档；
- 响应式电脑与手机页面；
- 部署所需配置说明。

首版不包含：

- 多助教或管理员角色；
- 家长直接登录；
- 学生文件上传和文档解析；
- 正式 AQA 分数预测；
- 复杂图表和营地级统计；
- 自动向家长发送反馈。

## 12. 测试与验收

### 功能验收

- 用户可登录并只看到自己的学生。
- 可新增、编辑和删除学生。
- 每名学生每天可保存一条记录，并能恢复自动保存草稿。
- 第二天生成时能准确引用前一天计划和进展。
- 可通过对话修改草稿且不丢失事实。
- 三种语言模式均可正确保存和查看。
- 不足 50 字或缺少下一步建议的反馈无法归档。
- AI 故障或保存失败不会清空输入。

### 安全验收

- 未登录用户无法读取业务数据。
- 用户不能通过修改 URL 或请求参数访问其他用户的数据。
- 浏览器源码和网络响应中不出现 DeepSeek API Key。

### 界面验收

- 桌面端保持三栏结构。
- 手机端内容按清晰顺序排列，无横向溢出。
- 1–10 名学生之间切换流畅。
- 保存、失败、自动保存和生成中状态均有明确反馈。

## 13. 部署

部署需要：

- 一个 Supabase 项目；
- 数据库迁移和 RLS 策略；
- 一个 DeepSeek API Key；
- 一个 Vercel 项目；
- Supabase URL、匿名公钥、服务端密钥和 DeepSeek Key 等环境变量；
- 首个登录账号；
- 部署后的回调 URL 和站点 URL 配置。

项目提供 `.env.example`、数据库迁移文件和部署说明，但不提交真实密钥。
